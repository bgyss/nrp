# Status Report — 2026-07-01

Scope: the three goal items delivered in this cycle (Mitsuba 3 exporter, OIDN
denoising, GPU benchmarking), plus an assessment of the overall state and measured
performance of the repository.

Hardware context for every number below: one Apple Silicon laptop (macOS), Python
3.12, PyTorch CPU/MPS builds, single process. Nothing here is quoted from the paper;
every figure was produced by a run in this repository. The paper's numbers were
measured on an NVIDIA RTX 5090 at production resolutions and are **not** directly
comparable.

## Goal item 1 — Mitsuba 3 exporter

**Delivered.** `nrp/mitsuba_exporter.py` (optional `mitsuba` extra) reproduces the
paper's §4.1 light-agnostic path pass over any Mitsuba scene XML or the built-in
cornell box: camera rays through the scene's own sensor, BSDF importance sampling with
no next-event estimation, throughput-based Russian roulette (paper-matching; can be
disabled for deterministic path counts), one cache segment per path segment with the
throughput *before* the segment, escape segments recorded with infinite t_max, and
first-hit G-buffer aux features (albedo from the BSDF's diffuse reflectance, depth,
shading normal, position) averaged over samples.

Measured: the 48×48 / 16 spp / 4-bounce cornell-box export produces **103,680 segments
in 1.6 s** (3.9 MB compressed). Scene emitters are ignored by construction — the pass
is light-agnostic, which is the paper's core requirement.

Verification: five gated unit tests (cache schema validation, exact per-pixel path
counts without RR, unit first-segment throughput, populated unit-length aux normals,
nonzero GATHERLIGHT for an in-scene emitter), plus the end-to-end training run below.

## Goal item 2 — OIDN denoising

**Delivered.** The paper's denoiser (Intel Open Image Denoise, RT filter, HDR mode,
albedo + normal guides) is now available behind the `denoise_image` dispatch
(`"denoise": {"method": "oidn"}` in a training config), with the dependency-free
joint bilateral filter remaining the default fallback. The `oidn` 0.2.x wrapper does
not expose the boolean parameter setter needed for HDR mode, so the implementation
calls `oidnSetFilter1b` on the already-loaded dylib via ctypes. On macOS the wheel
links against Homebrew's TBB (`brew install tbb` — installed on this machine during
the cycle).

Measured: on a synthetic HDR test (flat 2.0 signal + σ=0.5 Gaussian noise), OIDN cuts
MSE **0.248 → 0.0054 (46×)** while preserving the HDR mean (1.98 vs 2.0), confirming
the HDR flag reaches the filter.

Verification: gated unit tests (denoising quality threshold, HDR preservation,
dispatch equivalence, unknown-method error), plus the end-to-end pipeline below.

## Goal item 3 — GPU benchmarking

**Delivered.** `nrp/torch_backend/bench.py` measures full-frame proxy inference
latency across resolutions on every available device (cpu / mps / cuda) with warmup
and correct device synchronization (`torch.mps.synchronize` / `torch.cuda.synchronize`).

Measured (62,923-parameter sphere model):

| device | 48² | 128² | 256² | 512² | 1024² |
|---|---|---|---|---|---|
| cpu | 5.9 ms (170 Hz) | 14.1 ms (71 Hz) | 47.7 ms (21 Hz) | 110.6 ms (9.0 Hz) | 354 ms (2.8 Hz) |
| mps | 2.1 ms (481 Hz) | 2.8 ms (359 Hz) | 8.5 ms (117 Hz) | 26.8 ms (37 Hz) | 116 ms (8.6 Hz) |

Headline: **this laptop's GPU (MPS) sustains the paper's ~30–60 Hz interactive
relighting range up to 512×512** (37 Hz), a 4–13× speedup over CPU depending on
resolution. Reaching the paper's rates at production resolution would require CUDA +
tiny-cuda-nn, which remains documented future work.

Verification: unit tests for timing sanity and device discovery; results persisted to
`out/bench.json` by `mise run bench`.

## Combined end-to-end validation

The three items compose into the paper-exact data pipeline (§4.1 + §4.4): Mitsuba
path export → GATHERLIGHT targets → OIDN-denoised image pool → hashgrid-MLP training.
On the Mitsuba cornell box this reaches held-out **PSNR 25.87 dB vs raw GATHERLIGHT
(26.48 dB vs the denoised reference) in 48 s of CPU training** — the strongest quality
result in the repository, and notably better than the same architecture on the toy
tracer scene (19.17 dB), consistent with the cornell box being a better-conditioned
scene.

## State of the repository

- **Completeness:** every algorithmic component of the paper is implemented or
  explicitly substituted; see [paper-mapping.md](../paper-mapping.md). Remaining
  non-implemented areas: volumetric transport, fused GPU kernels
  (Triton/tiny-cuda-nn), compressed cache layout (fp16/rgb9e5), multi-view and
  per-layer NRPs, and production-scale scenes.
- **Testing:** 71 unit tests, all green (`mise run test`); optional-dependency tests
  skip cleanly when extras are absent. Ruff lint/format clean.
- **Toolchain:** nix flake (verified: `nix develop` → Python 3.12.13) + mise tasks +
  uv-managed venv; runtime deps are numpy + torch, with `mitsuba` and `oidn` extras.
- **Key quality results** (see [performance.md](../performance.md) for full tables):
  decoupling consistency 28.97 dB / 0.03% mean-radiance agreement against an
  independent re-trace; torch inverse optimization recovers a hidden light to center
  error 0.013 in a unit box (30× better than the numpy backend's naive optimizer,
  isolating the value of the paper's §5.3 reparameterization + tonemapped loss).

## Risks / caveats

- SMAPE remains ~0.9 on held-out lights across backends — dominated by near-zero
  pixels where both prediction and Monte Carlo reference are noisy; PSNR is the more
  informative aggregate at these sample counts.
- Recovered emission intensity in inverse runs sits ~7–10% low (proxy amplitude
  bias); geometry recovery is excellent. Worth revisiting with longer training.
- The scalar-Python Mitsuba export loop is fine at toy scale (1.6 s) but will not
  scale to the paper's 960×402 @ 512 spp scenes; a vectorized drjit loop is item 1 of
  the [roadmap](../roadmap.md).

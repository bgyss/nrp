# Status Report — 2026-07-02

Scope: the full ten-item [roadmap](../roadmap.md) is now complete. This report covers
the seven items delivered since [2026-07-01.md](2026-07-01.md)
(volumetric transport, GPU-fused GATHERLIGHT + MPS training, quad/multi-light
inverse optimization, packed §4.2 cache layout, paper-scale training, the Fig. 6
image-based baseline, multi-view NRPs, per-layer compositing, and the SSIM/FLIP
ablation suite — nine items across seven commits, since two shipped together), and
gives an overall before/after picture of the repository's measured capability.

Hardware context is unchanged: one Apple Silicon laptop (macOS), Python 3.12,
PyTorch CPU/MPS builds, single process, no CUDA hardware. Every number below was
produced by a run in this repository; none are quoted from the paper, whose figures
come from an RTX 5090 at production resolutions and are not directly comparable —
only scaling *shapes* are compared.

## Before/after: the repository's capability envelope

- **Quality ceiling.** The previous report's best result was 25.87 dB (Mitsuba
  cornell box, 3k iterations). Paper-scale training (item 6) now reaches **35.19 dB**
  on the same scene family — the ceiling was iteration- and capacity-bound, not
  architecture-bound, and there is no plateau yet at 50k iterations.
- **Data efficiency has an honest counter-result.** Item 9 is the one place this
  cycle where the paper's claim did not transfer: at matched supervision budget, a
  fixed 1,024-image dataset beat the rolling training pool by 1.83 dB. The cause was
  isolated (pool 64 caps concurrent light diversity; raising it to 256 recovers most
  of the gap) rather than left as an unexplained discrepancy.
- **Compactness, demonstrated three ways.** Packed caches (3.2–4.2×), multi-view
  proxies (0.76 MB for 3 views vs 10.5 MB of caches), and per-layer proxies (latency-
  neutral compositing) all land the paper's "cheap proxy replaces expensive cache"
  argument with concrete MB/ms numbers rather than assertion.
- **GPU path is now real, with a stated limit.** Gather and inference both show
  5–7× MPS speedups at scale, but end-to-end *training* wall-clock is 15–30% slower
  on MPS than CPU at this model size — under-filled batches, not a training defect —
  documented rather than hidden.
- **Perceptual metrics changed the story a plain PSNR table would have told.** In
  the ablation suite, the `none` (no aux, no encoding, no denoising) variant looks
  almost competitive on PSNR alone (23.8–25.8 dB); SSIM and FLIP are what expose it
  as a low-frequency lighting wash, confirming the paper's reason for reporting them.

## State of the repository

- **Completeness:** all ten roadmap items delivered; every algorithmic component of
  the paper is implemented or explicitly substituted — see
  [paper-mapping.md](../paper-mapping.md).
- **Testing:** 144 unit tests, all green (`mise run test`, 1 skip for an
  environment-gated optional path); ruff lint/format clean.
- **Toolchain:** unchanged — nix flake + mise tasks + uv-managed venv; `mitsuba` and
  `oidn` extras remain optional and skip cleanly when absent.
- **Known deviations from the paper** (see the README and CLAUDE.md tables): CPU/MPS
  PyTorch only (no tiny-cuda-nn/Triton fused kernel), OIDN optional with a bilateral
  fallback, softplus output head, Python-side path recording in the Mitsuba exporter.
  None of these block any roadmap item; they are the sources of the RTX-5090-vs-laptop
  gap noted throughout.

## Risks / caveats carried forward

- SMAPE remains ~0.6–0.9 across backends and scenes, dominated by near-zero pixels
  where the Monte Carlo reference itself is noisy at low spp; PSNR and the new
  SSIM/FLIP metrics are the more informative aggregates.
- The image-based-baseline result (item 9) means "just use a rolling pool" is not a
  free lunch at small pool sizes — anyone building on this code at toy scale should
  size the pool to the light diversity they need, not assume path-based training
  automatically wins.
- MPS training being slower than CPU at this parameter count is scale-dependent;
  it should invert once model width/batch size grow further (the paper-scale run
  already shows MPS pulling ahead of CPU by 1.6× at 8×256/50k, so the crossover is
  a function of iters/s not being profitable until models get some way to fill the
  GPU's parallelism).

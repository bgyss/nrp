Two weeks ago I found and fixed the bug behind a "perfect" 154.5 dB result
that was half fake — three neural light proxies silently outputting exactly
zero. That was rung 1 of a six-rung hardening pass through every caveat and
negative result my last project phase had produced. All six are done: two
negatives flipped into real fixes, a ~730x interactive-editing speedup,
one blocker honestly confirmed — and two times I caught the *same bug*
hiding again in code I'd just written.

**Re-earning the big claim.** Retrained the full 8-light rig with the fix
applied: all 8 proxies now produce real output — no more silent zeros. But
the additivity gate (proxy sum vs. ground truth) still misses preview-tier
quality (SSIM 0.73 vs. a 0.80 bar). Real progress, not a clean win, and I'm
reporting it that way.

**The bug found me twice more.** Checking the art-direction recovery by
hand instead of trusting my own automated pass/fail script, I found a light
whose "recovered" color was actually just the untouched starting guess — its
proxy output was technically nonzero, but only by five or six orders of
magnitude less than every other light, quietly under my own detection
threshold. Genuine recovery: 5 of 6 colorable lights, not the 6/6 my script
reported. Then, building a *new* "full retrain" benchmark for a real-scene
dynamic-geometry test, I caught myself reproducing rung 1's exact zero-
collapse mechanism in a from-scratch training loop I'd hand-rolled instead
of reusing the fixed code path. Same root cause, same fix, different file.
The lesson from two weeks ago about not trusting an aggregate metric applies
just as much to code you wrote five minutes ago.

**A negative, diagnosed all the way into a fix.** The two texture-conditioned
light proxies can't be helped by more training time or a bigger network —
4x the budget and 2x the width still left them 3–4 dB below the rig's
quality envelope. But the diagnosis (their texture was flattened into 192
raw input numbers with no spatial structure) pointed straight at the real
fix: the emission is *linear* in the texture — every texel is an
independent emitter — so the model now predicts a per-texel kernel and
contracts it with the texture at the output, instead of being asked to
learn that structure from scratch. Both proxies landed inside the quality
envelope (19.6/20.0 dB), and at a quarter of the training budget the new
head already beats every variant of the old one.

**An honest negative, worth stating plainly.** A toy-scale fix for
dynamic geometry that looked promising two months ago does not transfer to
a real, re-traced scene: neither tested repair strategy meets the recovery
target against a properly converged retrain — the closer one misses by
14.5 dB, the other by over 40 dB. Confirmed a real blocker, not softened one.

**Two things that shipped clean.** Storage cost for a rendered shot dropped
from 1.17x raw frame size to 0.59x by only keeping exact pixel data for a
subset of "approval" frames — zero quality cost on any frame tested. And
light-rig compositing now runs on the real WebGPU compute path in actual
Chrome, near-exact match against the CPU reference. The first measurement
was an honest miss (170 ms per slider nudge, re-running all 8 networks per
adjustment) — until noticing that a color nudge doesn't change any network
input at all. Caching each light's raw output and recompositing brings a
color adjustment to 1.3 ms at the 95th percentile (~730x the CPU path), and
even the worst case — editing a light's shape, which re-runs one full
network pass — lands at 30.9 ms. Both beat the 33 ms real-time stretch
target: the rig editor is now genuinely interactive.
